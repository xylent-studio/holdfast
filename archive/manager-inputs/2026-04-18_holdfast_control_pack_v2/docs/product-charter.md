# Holdfast Product Charter

## One-line definition
Holdfast is a personal operating system for staying in command of real life without turning life into project management.

## Product thesis
People do not usually lose control because they cannot create enough tasks. They lose control because capture is too slow, the day has no command view, deferral becomes disappearance, and continuity across devices is not trustworthy enough to relax.

Holdfast exists to fix that.

It should let the user:
- catch something immediately
- decide what matters now
- plan what does not belong now without losing it
- close the day cleanly
- pick back up on any device without friction
- trust that nothing important quietly vanished

## The feeling to preserve
Holdfast should feel:
- calm
- direct
- modern
- trustworthy
- private
- low-friction
- serious without being heavy

It should not feel:
- gamified
- managerial
- bureaucratic
- over-explained
- like a dashboard full of knobs
- like a developer tool
- like an AI narrating its own architecture

## What Holdfast is
Holdfast is:
- a command view for the day
- a fast capture tool
- a trusted place to park things without losing them
- a review surface for retrieval and pattern recognition
- a continuity layer across devices
- an offline-capable synced app that feels seamless

## What Holdfast is not
Holdfast is not:
- a generic task manager
- a team collaboration tool
- a kanban board
- a life-optimization game
- a notebook app with task features bolted on
- a settings-heavy system that asks the user to manage syncing, storage, or workflow internals
- a manual backup/import product disguised as modern software

## Core promise
Holdfast should make the user feel:
- I know what matters today.
- I am not carrying everything in my head.
- What I capture will still be there later.
- I can trust this app on any device.
- The app is helping me act, not asking me to manage the app.

## Primary product qualities
In order:
1. Trust
2. Speed
3. Clarity
4. Continuity
5. Restraint

If a feature improves capability but weakens trust, speed, clarity, continuity, or restraint, it is probably wrong.

## Non-negotiables
- Signed-in sync across devices is the normal experience.
- Offline use must work naturally and recover automatically.
- Fast capture beats feature density.
- The core spine stays simple: Now, Inbox, Upcoming, Review.
- Notes and tasks stay lightweight.
- Attachments and voice memos are first-class product features.
- The user should almost never need to think about sync mechanics, data durability, or storage internals.
- Settings should be minimal and meaningful.
- Product language must sound like a real app, not dev documentation.
- Every feature must earn its place.

## Product evolution principle
Do not let what Holdfast currently is hold back what it could or should become.

The prototype is evidence, not a cage.

Preserve:
- the product intent
- the trust model
- the command-oriented daily rhythm
- low-friction capture
- deliberate deferral instead of disappearance
- honest review and carry-forward
- the feeling that nothing important slips away

Replace without sentimentality if needed:
- one-file implementation constraints
- prototype storage assumptions
- awkward prototype flows
- labels chosen because they were easy to code
- helper text that explains the system instead of helping the user
- any structure that only made sense as scaffolding

## Product-truth test
When evaluating a change, do not ask:
- Does this match the prototype exactly?

Ask:
- Does this better express what Holdfast should become?
- Is this product truth or prototype accident?
- Does this increase trust?
- Does this make the app feel calmer, clearer, and more dependable?
- Would a strong product team be proud to ship this?
